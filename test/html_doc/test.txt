<div class="ltx_abstract">
<h6 class="ltx_title ltx_title_abstract">Abstract</h6>
<p class="ltx_p">Retrieval-augmented generation (RAG) has shown great potential for knowledge-intensive tasks, but its traditional architectures rely on static retrieval, limiting their effectiveness for complex questions that require sequential information-seeking. While agentic reasoning and search offer a more adaptive approach, most existing methods depend heavily on prompt engineering. In this work, we introduce RAG-Gym, a unified optimization framework that enhances information-seeking agents through fine-grained process supervision at each search step. We also propose ReSearch, a novel agent architecture that synergizes answer reasoning and search query generation within the RAG-Gym framework. Experiments on four challenging datasets show that RAG-Gym improves performance by up to 25.6% across various agent architectures, with ReSearch consistently outperforming existing baselines. Further analysis highlights the effectiveness of advanced LLMs as process reward judges and the transferability of trained reward models as verifiers for different LLMs. Additionally, we examine the scaling properties of training and inference in agentic RAG.
The project homepage is available at <a target="_blank" href="https://rag-gym.github.io/" title="" class="ltx_ref ltx_url ltx_font_typewriter">https://rag-gym.github.io/</a>.</p>

<section id="S1" class="ltx_section">
<h2 class="ltx_title ltx_title_section">
<span class="ltx_tag ltx_tag_section">1 </span>Introduction</h2>

<div id="S1.p1" class="ltx_para">
<p class="ltx_p">Large language models (LLMs) have revolutionized online information-seeking by providing direct responses to user needs. Such capabilities are often evaluated by knowledge-intensive tasks such as multi-hop question answering (QA), which require sequential reasoning over relevant knowledge <cite class="ltx_cite ltx_citemacro_citep">(Lewis etÂ al., <a href="#bib.bib23" title="" class="ltx_ref">2020</a>; Welbl etÂ al., <a href="#bib.bib54" title="" class="ltx_ref">2018</a>; Tang &amp; Yang, <a href="#bib.bib41" title="" class="ltx_ref">2024</a>)</cite>. Pre-trained on trillions of tokens from diverse sources, LLMs have demonstrated strong reasoning capabilities and acquired considerable general knowledge <cite class="ltx_cite ltx_citemacro_citep">(Wei etÂ al., <a href="#bib.bib51" title="" class="ltx_ref">2022a</a>)</cite>. As such, they are often used as standalone systems that process user queries and generate responses without relying on external utilities <cite class="ltx_cite ltx_citemacro_citep">(Brown etÂ al., <a href="#bib.bib3" title="" class="ltx_ref">2020</a>; Achiam etÂ al., <a href="#bib.bib1" title="" class="ltx_ref">2023</a>; Touvron etÂ al., <a href="#bib.bib42" title="" class="ltx_ref">2023a</a>, <a href="#bib.bib43" title="" class="ltx_ref">b</a>)</cite>. Despite these impressive capabilities, LLMs often struggle with questions where they may lack sufficient and up-to-date domain knowledge. This may lead to inaccurate responses or hallucinations, which can be particularly problematic in high-stakes fields <cite class="ltx_cite ltx_citemacro_citep">(Zhang etÂ al., <a href="#bib.bib67" title="" class="ltx_ref">2023b</a>; Sahoo etÂ al., <a href="#bib.bib33" title="" class="ltx_ref">2024</a>; Ji etÂ al., <a href="#bib.bib15" title="" class="ltx_ref">2023</a>)</cite>.</p>
</div>
<div id="S1.p2" class="ltx_para">
<p class="ltx_p">Retrieval-augmented generation (RAG) has emerged as a promising solution to address knowledge-intensive tasks <cite class="ltx_cite ltx_citemacro_citep">(Lewis etÂ al., <a href="#bib.bib23" title="" class="ltx_ref">2020</a>)</cite>. By grounding the LLM outputs in relevant information returned by information retrieval (IR) systems, RAG improves both accuracy and verifiability <cite class="ltx_cite ltx_citemacro_citep">(Gao etÂ al., <a href="#bib.bib10" title="" class="ltx_ref">2023</a>)</cite>. Expanding on this paradigm, information-seeking agents such as ReAct <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>)</cite> iteratively interact with IR systems to tackle complex questions more effectively <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>; Asai etÂ al., <a href="#bib.bib2" title="" class="ltx_ref">2024</a>; Shinn etÂ al., <a href="#bib.bib37" title="" class="ltx_ref">2024</a>; Li etÂ al., <a href="#bib.bib25" title="" class="ltx_ref">2025</a>)</cite>. However, most existing methods rely heavily on prompting techniques <cite class="ltx_cite ltx_citemacro_citep">(Trivedi etÂ al., <a href="#bib.bib44" title="" class="ltx_ref">2023</a>; Asai etÂ al., <a href="#bib.bib2" title="" class="ltx_ref">2024</a>; Jiang etÂ al., <a href="#bib.bib16" title="" class="ltx_ref">2023</a>; Press etÂ al., <a href="#bib.bib29" title="" class="ltx_ref">2023</a>)</cite>, which require substantial efforts for the prompt design and might not generalize well across different tasks <cite class="ltx_cite ltx_citemacro_citep">(Khot etÂ al., <a href="#bib.bib21" title="" class="ltx_ref">2023</a>; Shao etÂ al., <a href="#bib.bib35" title="" class="ltx_ref">2023</a>; Yu etÂ al., <a href="#bib.bib62" title="" class="ltx_ref">2024</a>)</cite>.</p>
</div>

<div id="S1.p3" class="ltx_para">
<p class="ltx_p">To improve such agents in solving knowledge-intensive and multi-hop QA tasks, we propose <span class="ltx_text ltx_font_bold">RAG-Gym</span>, a unified framework for optimizing agentic RAG through process supervision. RAG-Gym formulates the knowledge-intensive QA as a nested Markov Decision Process (MDP), where the outer MDP action quality is annotated with process rewards, which in turn guide the inner MDP token generation. By supporting diverse agent architectures and process supervision methods, RAG-Gym provides a comprehensive platform for improving information-seeking agents.
In addition to RAG-Gym, we propose <span class="ltx_text ltx_font_bold">ReSearch</span>, a novel agent architecture that unifies answer reasoning and search query generation in a single action. This design explicitly aligns the generated query with missing information in answer construction.
Through extensive experiments, we demonstrate that RAG-Gym effectively enhances agentic RAG performance on knowledge-intensive tasks, with ReSearch outperforming other baseline agents. Notably, our findings highlight the benefits of training process reward models as verifiers for agentic RAG actions and show their transferability across different LLMs. Further analysis reveals the effectiveness of advanced LLMs as process reward judges and uncovers the scaling laws of process supervision in RAG-Gym across both training and inference phases.</p>
</div>
<div id="S1.p4" class="ltx_para">
<p class="ltx_p">Our key contributions are four-fold: (1) We introduce RAG-Gym, a unified framework for optimizing agentic RAG with process supervision. (2) We propose ReSearch, a novel agent architecture that synergizes answer reasoning and search, achieving state-of-the-art performance over existing baselines. (3) We demonstrate that using trained process reward models as verifiers significantly improves search agent performance. (4) We provide a comprehensive analysis of process supervision sources, reward model transferability, and scaling laws in agentic RAG.</p>
</div>
</section>
<section id="S2" class="ltx_section">
<h2 class="ltx_title ltx_title_section">
<span class="ltx_tag ltx_tag_section">2 </span>RAG-Gymnasium (RAG-Gym)</h2>

<div id="S2.p1" class="ltx_para">
<p class="ltx_p">RAG-Gym models knowledge-intensive QA as a nested Markov Decision Process (MDP), where the outer MDP governs high-level action generation through interactions with the information retrieval (IR) environment, while the inner MDP controls token generation within LLMs. This formulation is compatible with various agentic RAG architecture that leverages IR for augmented answer generation. By assigning process rewards from high-level actions to sequential token generation, RAG-Gym enables effective tuning of language agents, which can be further utilized for optimizing agent parameters or training process verifiers.</p>
</div>

<p class="ltx_p"><span class="ltx_text ltx_font_bold">Action Space <math id="S2.SS1.p3.m1" class="ltx_Math" alttext="\mathcal{A}" display="inline"><semantics><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><annotation encoding="application/x-tex">\mathcal{A}</annotation></semantics></math>.</span> While different agents may use various designs to reason about the given state and generate token sequences, we standardize these action sequences by defining a shared abstract action space <math id="S2.SS1.p3.m2" class="ltx_Math" alttext="\mathcal{A}" display="inline"><semantics><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><annotation encoding="application/x-tex">\mathcal{A}</annotation></semantics></math>. At each time step <math id="S2.SS1.p3.m3" class="ltx_Math" alttext="t" display="inline"><semantics><mi>t</mi><annotation encoding="application/x-tex">t</annotation></semantics></math>, the action <math id="S2.SS1.p3.m4" class="ltx_Math" alttext="a_{t}" display="inline"><semantics><msub><mi>a</mi><mi>t</mi></msub><annotation encoding="application/x-tex">a_{t}</annotation></semantics></math> can be either a search query or a predicted answer to the original question. Thus, we define the action space <math id="S2.SS1.p3.m5" class="ltx_Math" alttext="\mathcal{A}=\mathcal{A}_{q}\cup\mathcal{A}_{p}" display="inline"><semantics><mrow><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mo>=</mo><mrow><msub><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mi>q</mi></msub><mo>âˆª</mo><msub><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mi>p</mi></msub></mrow></mrow><annotation encoding="application/x-tex">\mathcal{A}=\mathcal{A}_{q}\cup\mathcal{A}_{p}</annotation></semantics></math> where <math id="S2.SS1.p3.m6" class="ltx_Math" alttext="\mathcal{A}_{q}" display="inline"><semantics><msub><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mi>q</mi></msub><annotation encoding="application/x-tex">\mathcal{A}_{q}</annotation></semantics></math> is the set of all possible queries, and <math id="S2.SS1.p3.m7" class="ltx_Math" alttext="\mathcal{A}_{p}" display="inline"><semantics><msub><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mi>p</mi></msub><annotation encoding="application/x-tex">\mathcal{A}_{p}</annotation></semantics></math> denotes the set of possible answers.</p>
</div>
<div id="S2.SS1.p4" class="ltx_para ltx_noindent">
<p class="ltx_p"><span class="ltx_text ltx_font_bold">IR Environment.</span> The environment of the outer MDP in RAG-Gym is driven by an IR system, which takes a search query <math id="S2.SS1.p4.m1" class="ltx_Math" alttext="q_{t}" display="inline"><semantics><msub><mi>q</mi><mi>t</mi></msub><annotation encoding="application/x-tex">q_{t}</annotation></semantics></math> as input and returns a set of relevant documents <math id="S2.SS1.p4.m2" class="ltx_Math" alttext="{D}_{t}" display="inline"><semantics><msub><mi>D</mi><mi>t</mi></msub><annotation encoding="application/x-tex">{D}_{t}</annotation></semantics></math> as output. Formally, the IR system can be represented as a mapping from <math id="S2.SS1.p4.m3" class="ltx_Math" alttext="\mathcal{A}_{q}" display="inline"><semantics><msub><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mi>q</mi></msub><annotation encoding="application/x-tex">\mathcal{A}_{q}</annotation></semantics></math> to <math id="S2.SS1.p4.m4" class="ltx_Math" alttext="\mathcal{P}(\mathcal{D})" display="inline"><semantics><mrow><mi class="ltx_font_mathcaligraphic">ğ’«</mi><mo lspace="0em" rspace="0em">â€‹</mo><mrow><mo stretchy="false">(</mo><mi class="ltx_font_mathcaligraphic">ğ’Ÿ</mi><mo stretchy="false">)</mo></mrow></mrow><annotation encoding="application/x-tex">\mathcal{P}(\mathcal{D})</annotation></semantics></math>
where <math id="S2.SS1.p4.m5" class="ltx_Math" alttext="\mathcal{P}(\mathcal{D})" display="inline"><semantics><mrow><mi class="ltx_font_mathcaligraphic">ğ’«</mi><mo lspace="0em" rspace="0em">â€‹</mo><mrow><mo stretchy="false">(</mo><mi class="ltx_font_mathcaligraphic">ğ’Ÿ</mi><mo stretchy="false">)</mo></mrow></mrow><annotation encoding="application/x-tex">\mathcal{P}(\mathcal{D})</annotation></semantics></math> is the power set of <math id="S2.SS1.p4.m6" class="ltx_Math" alttext="\mathcal{D}" display="inline"><semantics><mi class="ltx_font_mathcaligraphic">ğ’Ÿ</mi><annotation encoding="application/x-tex">\mathcal{D}</annotation></semantics></math>. The retrieval process is determined by the underlying text retriever and specific retrieval settings (e.g., the number of returned documents). These settings are fixed to ensure stability of state transitions.</p>
</div>
<div id="S2.SS1.p5" class="ltx_para ltx_noindent">
<p class="ltx_p"><span class="ltx_text ltx_font_bold">MDP Workflow.</span> For a given question <math id="S2.SS1.p5.m1" class="ltx_Math" alttext="\mathcal{Q}" display="inline"><semantics><mi class="ltx_font_mathcaligraphic">ğ’¬</mi><annotation encoding="application/x-tex">\mathcal{Q}</annotation></semantics></math>, the MDP starts at the initial state <math id="S2.SS1.p5.m2" class="ltx_Math" alttext="s_{1}=(\mathcal{Q},\emptyset)" display="inline"><semantics><mrow><msub><mi>s</mi><mn>1</mn></msub><mo>=</mo><mrow><mo stretchy="false">(</mo><mi class="ltx_font_mathcaligraphic">ğ’¬</mi><mo>,</mo><mi mathvariant="normal">âˆ…</mi><mo stretchy="false">)</mo></mrow></mrow><annotation encoding="application/x-tex">s_{1}=(\mathcal{Q},\emptyset)</annotation></semantics></math>. For every step <math id="S2.SS1.p5.m3" class="ltx_Math" alttext="t" display="inline"><semantics><mi>t</mi><annotation encoding="application/x-tex">t</annotation></semantics></math>, <math id="S2.SS1.p5.m4" class="ltx_Math" alttext="a_{t}" display="inline"><semantics><msub><mi>a</mi><mi>t</mi></msub><annotation encoding="application/x-tex">a_{t}</annotation></semantics></math> is sampled from the agentâ€™s policy <math id="S2.SS1.p5.m5" class="ltx_math_unparsed" alttext="\pi_{f(\theta)}(\cdot\,|\,s_{t})" display="inline"><semantics><mrow><msub><mi>Ï€</mi><mrow><mi>f</mi><mo lspace="0em" rspace="0em">â€‹</mo><mrow><mo stretchy="false">(</mo><mi>Î¸</mi><mo stretchy="false">)</mo></mrow></mrow></msub><mrow><mo stretchy="false">(</mo><mo lspace="0em">â‹…</mo><mo fence="false" rspace="0.337em" stretchy="false">|</mo><msub><mi>s</mi><mi>t</mi></msub><mo stretchy="false">)</mo></mrow></mrow><annotation encoding="application/x-tex">\pi_{f(\theta)}(\cdot\,|\,s_{t})</annotation></semantics></math> where <math id="S2.SS1.p5.m6" class="ltx_Math" alttext="\pi_{f(\theta)}:\mathcal{S}\rightarrow\Delta(\mathcal{A})" display="inline"><semantics><mrow><msub><mi>Ï€</mi><mrow><mi>f</mi><mo lspace="0em" rspace="0em">â€‹</mo><mrow><mo stretchy="false">(</mo><mi>Î¸</mi><mo stretchy="false">)</mo></mrow></mrow></msub><mo lspace="0.278em" rspace="0.278em">:</mo><mrow><mi class="ltx_font_mathcaligraphic">ğ’®</mi><mo stretchy="false">â†’</mo><mrow><mi mathvariant="normal">Î”</mi><mo lspace="0em" rspace="0em">â€‹</mo><mrow><mo stretchy="false">(</mo><mi class="ltx_font_mathcaligraphic">ğ’œ</mi><mo stretchy="false">)</mo></mrow></mrow></mrow></mrow><annotation encoding="application/x-tex">\pi_{f(\theta)}:\mathcal{S}\rightarrow\Delta(\mathcal{A})</annotation></semantics></math> defines the action distribution given a state. <section id="S2.SS2" class="ltx_subsection">
<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">2.2 </span>Improving Search Agents with Process Supervision</h3>

<div id="S2.SS2.p1" class="ltx_para">
<p class="ltx_p">While outcome-based rewards, such as answer correctness, provide a clear optimization objective, they offer limited insight into the quality of intermediate reasoning and search steps. In RAG-Gym, a poorly formulated query may still lead to a correct answer if the language agent compensates with internal knowledge, making it difficult to assess the effectiveness of the search query. To address this, process supervision is essential. Recent advancements in process reward modeling <cite class="ltx_cite ltx_citemacro_citep">(Lightman etÂ al., <a href="#bib.bib26" title="" class="ltx_ref">2024</a>; Wang etÂ al., <a href="#bib.bib48" title="" class="ltx_ref">2024a</a>)</cite> show that guiding intermediate steps enhances both performance and robustness in language agents. Since outer MDP actions in RAG-Gym are discrete, optimizing agents solely through outcome rewards is challenging. By incorporating process rewards, RAG-Gym enables more effective LLM tuning, aligning token generation with high-quality search behavior. Section <a href="#S2.SS2.SSS1" title="2.2.1 Collection of Process Reward Data â€£ 2.2 Improving Search Agents with Process Supervision â€£ 2 RAG-Gymnasium (RAG-Gym) â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">2.2.1</span></a> details our process reward data collection pipeline, and Section <a href="#S2.SS2.SSS2" title="2.2.2 Tuning Agents with Process Supervision â€£ 2.2 Improving Search Agents with Process Supervision â€£ 2 RAG-Gymnasium (RAG-Gym) â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">2.2.2</span></a> describes the algorithms for tuning language agents with process supervision.</p>
</div>
<section id="S2.SS2.SSS1" class="ltx_subsubsection">
<h4 class="ltx_title ltx_title_subsubsection">
<span class="ltx_tag ltx_tag_subsubsection">2.2.1 </span>Collection of Process Reward Data</h4>

<div id="S2.SS2.SSS1.p1" class="ltx_para">
<p class="ltx_p">To evaluate intermediate reasoning and search steps in RAG-Gym, we design a process reward function that assesses queries based on three key criteria:</p>
<ul id="S2.I1" class="ltx_itemize">
<li id="S2.I1.i1" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="S2.I1.i1.p1" class="ltx_para">
<p class="ltx_p">Sufficiency: A query must first be necessary. If the retrieval history already contains sufficient information, answering should be the preferred action instead of searching further.</p>
</div>
</li>
<li id="S2.I1.i2" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="S2.I1.i2.p1" class="ltx_para">
<p class="ltx_p">Utility: Queries should also be precise, actionable, and foundational to solving the question while avoiding unnecessary details.</p>
</div>
</li>
<li id="S2.I1.i3" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="S2.I1.i3.p1" class="ltx_para">
<p class="ltx_p">Redundancy: Queries should introduce new, useful information rather than repeating past searches.</p>
</div>
</li>
</ul>
<p class="ltx_p">These criteria ensure that queries are efficient, targeted, and contribute meaningfully to constructing the final answer.</p>
</div>
<div id="S2.SS2.SSS1.p2" class="ltx_para">
<p class="ltx_p">The data collection pipeline begins with trajectory sampling, where the language agent generates a sequence of actions based on its current policy. At each step in the trajectory, multiple candidate actions are proposed, and the best action is selected according to predefined evaluation criteria. To streamline the annotation process and ensure consistency, we employ a ranking-based evaluation framework rather than assigning numerical scores. The selected action is then executed, and the trajectory transitions to the next state. This process is repeated iteratively until the trajectory terminates.</p>
</div>
<div id="S2.SS2.SSS1.p3" class="ltx_para">
<p class="ltx_p">To ensure quality, only trajectories that result in a correct final answer are retained, as determined by the outcome reward. This filtering guarantees that the selected actions not only align with the process reward criteria but also contribute to successful task completion. To address the challenges of slow and costly human annotation, we leverage LLMs such as GPT-4o to annotate the sampled trajectories. As demonstrated in our experiments (Table <a href="#S5.T2" title="Table 2 â€£ 5.1 Comparison of Different Reward Sources â€£ 5 Analysis and Discussion â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">2</span></a>), annotations generated by GPT-4o exhibit high reliability, closely aligning with domain expert judgments. This approach enables scalable and efficient data collection, making it feasible to gather high-quality process reward data at scale.</p>
</div>
</section>
<section id="S2.SS2.SSS2" class="ltx_subsubsection">
<h4 class="ltx_title ltx_title_subsubsection">

<div id="S2.SS2.SSS2.p1" class="ltx_para">

</div>

<section id="S3" class="ltx_section">
<h2 class="ltx_title ltx_title_section">
<span class="ltx_tag ltx_tag_section">3 </span>Reasoning and Search (ReSearch) Agent</h2>

<div id="S3.p1" class="ltx_para">
<p class="ltx_p">Existing agent architectures, such as ReAct <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>)</cite>, exhibit inherent limitations when applied to knowledge-intensive tasks. They can only reason about either the answer generation or the query generation at each step. This may lead to queries that cannot directly contribute to answer construction, as these two tasks are not necessarily aligned during reasoning. As illustrated in Appendix <a href="#A5" title="Appendix E Case Studies â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">E</span></a>, existing designs may generate queries that fail to retrieve the most relevant evidence, leading to suboptimal information acquisition and degrading answer quality. This limitation underscores the necessity of an architecture that explicitly aligns search queries with answer construction.</p>

<p class="ltx_p">where <math id="S3.p2.m5" class="ltx_Math" alttext="m_{i}" display="inline"><semantics><msub><mi>m</mi><mi>i</mi></msub><annotation encoding="application/x-tex">m_{i}</annotation></semantics></math> represents the summarized answer to <math id="S3.p2.m6" class="ltx_Math" alttext="q_{i}" display="inline"><semantics><msub><mi>q</mi><mi>i</mi></msub><annotation encoding="application/x-tex">q_{i}</annotation></semantics></math> given <math id="S3.p2.m7" class="ltx_Math" alttext="D_{i}" display="inline"><semantics><msub><mi>D</mi><mi>i</mi></msub><annotation encoding="application/x-tex">D_{i}</annotation></semantics></math>. This summarization step filters out irrelevant information and mitigates the challenge of long-context processing, allowing the agent to focus on the most relevant facts when constructing an answer.</p>
</div>
<div id="S3.p3" class="ltx_para">
<p class="ltx_p">Using this refined knowledge <math id="S3.p3.m1" class="ltx_Math" alttext="\mathcal{H}_{t}^{\prime}" display="inline"><semantics><msubsup><mi class="ltx_font_mathcaligraphic">â„‹</mi><mi>t</mi><mo>â€²</mo></msubsup><annotation encoding="application/x-tex">\mathcal{H}_{t}^{\prime}</annotation></semantics></math>, the agent then engages in structured reasoning to infer a candidate answer to the question.
It will then examine the reasoning steps and judge if all claims are well grounded in the history.
If the agent determines that all claims in the answer reasoning are supported by retrieved evidence, it outputs the answer as its final action. Otherwise, it identifies unverified claims, which are statements that lack sufficient justification based on the available evidence. These unverified claims serve as the basis for generating the next search query, which is specifically designed to retrieve the missing information. The retrieved documents from this query are then added to <math id="S3.p3.m2" class="ltx_Math" alttext="\mathcal{H}_{t}" display="inline"><semantics><msub><mi class="ltx_font_mathcaligraphic">â„‹</mi><mi>t</mi></msub><annotation encoding="application/x-tex">\mathcal{H}_{t}</annotation></semantics></math>, and the reasoning process is repeated iteratively until all claims are verified or the retrieval budget is exhausted.</p>
</div>
<div id="S3.p4" class="ltx_para">
<p class="ltx_p">A key advantage of the ReSearch agent lies in its answer-driven query formulation, which ensures that each search query is explicitly designed to search for missing information in the answer reasoning process. In contrast, conventional architectures such as ReAct rely on implicit LLM heuristics to determine when and how to issue search queries, often resulting in the retrieval of unhelpful information that cannot be effectively utilized in answer generation. By structuring query generation around ungrounded claims, our approach guarantees that retrieved evidence directly contributes to constructing the final answer, significantly improving retrieval efficiency and overall response quality. Empirical results, as presented in Section <a href="#S4" title="4 Results â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">4</span></a>, demonstrate that this structured approach to query generation leads to more precise and well-supported answers compared to conventional information-seeking agents.</p>
</div>
</section>

<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">4.1 </span>Experimental Settings</h3>

<div id="S4.SS1.p1" class="ltx_para">
<p class="ltx_p">To evaluate how different agents perform on knowledge-intensive QA tasks and how they can benefit from various process supervision methods implemented in RAG-Gym, we select different datasets that are both knowledge-intensive and reasoning-intensive, covering questions from both general and medical domains. Specifically, we choose HotpotQA <cite class="ltx_cite ltx_citemacro_citep">(Yang etÂ al., <a href="#bib.bib60" title="" class="ltx_ref">2018</a>)</cite>, 2WikiMultihopQA <cite class="ltx_cite ltx_citemacro_citep">(Ho etÂ al., <a href="#bib.bib11" title="" class="ltx_ref">2020</a>)</cite>, and Bamboogle <cite class="ltx_cite ltx_citemacro_citep">(Press etÂ al., <a href="#bib.bib29" title="" class="ltx_ref">2023</a>)</cite>, which are commonly used multi-hop QA datasets constructed from Wikipedia. We also include the MedQA dataset <cite class="ltx_cite ltx_citemacro_citep">(Jin etÂ al., <a href="#bib.bib17" title="" class="ltx_ref">2021</a>)</cite>, which consists of medical exam questions requiring specialized domain knowledge and complex reasoning. Following prior work <cite class="ltx_cite ltx_citemacro_citep">(Shao etÂ al., <a href="#bib.bib35" title="" class="ltx_ref">2023</a>; Xiong etÂ al., <a href="#bib.bib57" title="" class="ltx_ref">2024</a>)</cite>, we use Exact Match (EM) and F1 scores as the evaluation metrics of HotpotQA, 2WikiMultihopQA, and Bamboogle. The multi-choice MedQA dataset is evaluated with accuracy (Acc) as the metric. We also computed the average EM and F1 scores across different tasks, with accuracy being considered as both EM and F1 scores in the multi-choice evaluation setting.</p>
</div>
<div id="S4.SS1.p2" class="ltx_para">
<p class="ltx_p">We tested the RAG-Gym process supervision methods on six different agent designs, including the direct prompting (Direct), Chain-of-thought (CoT) <cite class="ltx_cite ltx_citemacro_citep">(Wei etÂ al., <a href="#bib.bib52" title="" class="ltx_ref">2022b</a>)</cite>, RAG <cite class="ltx_cite ltx_citemacro_citep">(Lewis etÂ al., <a href="#bib.bib23" title="" class="ltx_ref">2020</a>)</cite>, ReAct <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>)</cite>, Search-o1 <cite class="ltx_cite ltx_citemacro_citep">(Li etÂ al., <a href="#bib.bib25" title="" class="ltx_ref">2025</a>)</cite>, and the newly proposed ReSearch. More descriptions about the baselines can be found in Appendix <a href="#A3" title="Appendix C Baseline Descriptions â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">C</span></a>.
We selected Llama-3.1-8B-Instruct <cite class="ltx_cite ltx_citemacro_citep">(Dubey etÂ al., <a href="#bib.bib7" title="" class="ltx_ref">2024</a>)</cite> as the base LLM for the implementations of various reasoning and search agents, due to its context length of 128k tokens and its availability of open-source parameters. We also involved GPT-4o-mini as the representative of proprietary LLMs to examine if the trained reward models can be generalized to guide the action selection of other LLMs.
For the implementation of process supervision algorithms in RAG-Gym, we sampled 1k instances from the training sets of both HotpotQA and MedQA to generate process reward data for LLM tuning. 2WikiMultihopQA and Bamboogle were tested using LLMs trained on HotpotQA to see if the tuned agents are generalizable to other datasets with similar formats. More implementation details can be found in Appendix <a href="#A4" title="Appendix D Implementation Details â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">D</span></a>.</p>

<p class="ltx_p">The results also show that ReSearch consistently outperforms other agents, both in the ZSL setting and in settings with process supervision. Without tuning, ReSearch achieves strong zero-shot performance, demonstrating the effectiveness of explicitly aligning answer reasoning with query generation. Using process reward models, ReSearch achieves state-of-the-art performance, with an average EM score of 54.31% and an average F1 score of 62.41% across different datasets. Furthermore, ReSearch exhibits superior generalization, achieving top scores on 2WikiMultihopQA and Bamboogle without task-specific fine-tuning. These results validate the ability of ReSearchâ€™s unified reasoning and search framework to effectively leverage process supervision for both task-specific improvements and broader generalization across knowledge-intensive QA tasks.</p>

<div id="S5.SS2.p1" class="ltx_para">
<p class="ltx_p">For the evaluation of training sample size and its impacts on the performance of ReSearch agents, we conducted experiments using process reward models trained on varying numbers of instances, ranging from 250 to 1000 questions. The results, presented in Figure <a href="#S4.F4" title="Figure 4 â€£ 4.4 Reward Model Transferability â€£ 4 Results â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">4</span></a>, show how the agentâ€™s performance scales with the availability of more training data across four datasets.
In general, the performance of ReSearch improves with an increasing number of training samples, but the gains tend to converge as the sample size grows. Notably, there is a sharp improvement in F1 scores on HotpotQA, 2WikiMultihopQA, and Bamboogle when comparing the ZSL baseline to process reward models trained on 250 samples, showing that even a small amount of process reward data can yield significant performance gains. As the sample size increases, however, the performance gradually recovers and eventually surpasses the ZSL baseline, achieving the highest accuracy of 71.72 with 1000 samples. This underscores the importance of sufficient training data in capturing the nuanced reasoning and query-generation processes required for specialized tasks.</p>

<div id="S5.SS3.p1" class="ltx_para">
<p class="ltx_p">Since process reward models optimize action-taking by identifying high-quality actions from the generated candidates during inference, we explored how the agent performance changes with the increasing number of sampled actions at each time step. Specifically, for HotpotQA and Bamboogle, the F1 score continues to rise as more actions are sampled, demonstrating the benefits of expanding the candidate set to enable better action selection at each step. However, performance gains gradually diminish, indicating that the agent reaches a point where additional sampled actions contribute less to improvement. This suggests that while action sampling is beneficial, there is a limit to how much additional sampling enhances decision-making.</p>

<section id="S6.SS1" class="ltx_subsection">
<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">6.1 </span>Retrieval-Augmented Generation (RAG)</h3>

<div id="S6.SS1.p1" class="ltx_para">
<p class="ltx_p">RAG has emerged as a powerful paradigm for enhancing language models in knowledge-intensive tasks. By integrating external knowledge sources into generative models, RAG enables the production of more accurate and informative responses. This approach has seen success across various domains, such as open-domain question answering <cite class="ltx_cite ltx_citemacro_citep">(Karpukhin etÂ al., <a href="#bib.bib20" title="" class="ltx_ref">2020</a>)</cite> and conversational agents <cite class="ltx_cite ltx_citemacro_citep">(Shuster etÂ al., <a href="#bib.bib38" title="" class="ltx_ref">2021</a>)</cite>. Researchers have aimed to improve RAG by optimizing retrieval components <cite class="ltx_cite ltx_citemacro_citep">(Zhang etÂ al., <a href="#bib.bib65" title="" class="ltx_ref">2023a</a>; Nguyen etÂ al., <a href="#bib.bib28" title="" class="ltx_ref">2024</a>; Xu etÂ al., <a href="#bib.bib59" title="" class="ltx_ref">2024b</a>)</cite> or enhancing the generative modelâ€™s ability to leverage retrieved content <cite class="ltx_cite ltx_citemacro_citep">(Izacard etÂ al., <a href="#bib.bib14" title="" class="ltx_ref">2023</a>; Zhang etÂ al., <a href="#bib.bib66" title="" class="ltx_ref">2024b</a>; Fang etÂ al., <a href="#bib.bib8" title="" class="ltx_ref">2024</a>; Wei etÂ al., <a href="#bib.bib53" title="" class="ltx_ref">2024</a>; Jin etÂ al., <a href="#bib.bib19" title="" class="ltx_ref">2024</a>; Wang etÂ al., <a href="#bib.bib50" title="" class="ltx_ref">2024c</a>; Zhang etÂ al., <a href="#bib.bib64" title="" class="ltx_ref">2025</a>)</cite>. Applications of RAG extend to specialized fields, like biomedical question answering <cite class="ltx_cite ltx_citemacro_citep">(Xiong etÂ al., <a href="#bib.bib57" title="" class="ltx_ref">2024</a>; Xu etÂ al., <a href="#bib.bib58" title="" class="ltx_ref">2024a</a>; Kim etÂ al., <a href="#bib.bib22" title="" class="ltx_ref">2024</a>)</cite>, showcasing its versatility. Despite these advancements, vanilla RAG architectures typically rely on a single round of retrieval, limiting their effectiveness in scenarios requiring iterative reasoning and complex information integration.</p>
</div>
</section>
<section id="S6.SS2" class="ltx_subsection">
<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">6.2 </span>Multi-hop Question Answering (QA)</h3>

<div id="S6.SS2.p1" class="ltx_para">
<p class="ltx_p">Multi-hop QA tasks require systems to perform reasoning across multiple pieces of information, often from disparate sources, to derive a correct answer <cite class="ltx_cite ltx_citemacro_citep">(Yang etÂ al., <a href="#bib.bib60" title="" class="ltx_ref">2018</a>; Ho etÂ al., <a href="#bib.bib11" title="" class="ltx_ref">2020</a>)</cite>. These tasks are inherently challenging for traditional RAG architectures due to their single retrieval pass, which restricts the ability to gather all necessary information for complex questions. Early attempts to address multi-hop QA in RAG systems include agent designs such as ReAct <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>)</cite>, which integrate reasoning processes into retrieval steps <cite class="ltx_cite ltx_citemacro_citep">(Trivedi etÂ al., <a href="#bib.bib44" title="" class="ltx_ref">2023</a>; Asai etÂ al., <a href="#bib.bib2" title="" class="ltx_ref">2024</a>; Jiang etÂ al., <a href="#bib.bib16" title="" class="ltx_ref">2023</a>; Press etÂ al., <a href="#bib.bib29" title="" class="ltx_ref">2023</a>; Ram etÂ al., <a href="#bib.bib31" title="" class="ltx_ref">2023</a>; Wang etÂ al., <a href="#bib.bib49" title="" class="ltx_ref">2024b</a>; Khot etÂ al., <a href="#bib.bib21" title="" class="ltx_ref">2023</a>; Shao etÂ al., <a href="#bib.bib35" title="" class="ltx_ref">2023</a>; Wang etÂ al., <a href="#bib.bib47" title="" class="ltx_ref">2023</a>; Shi etÂ al., <a href="#bib.bib36" title="" class="ltx_ref">2024</a>; Yu etÂ al., <a href="#bib.bib62" title="" class="ltx_ref">2024</a>)</cite>. Nevertheless, these strategies commonly rely on finely tuned heuristics or handcrafted prompts, lacking robustness when facing diverse question formulations.</p>
</div>
</section>
<section id="S6.SS3" class="ltx_subsection">
<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">6.3 </span>Process Reward Modeling (PRM)</h3>

<div id="S6.SS3.p1" class="ltx_para">
<p class="ltx_p">PRM provides a structured framework for guiding LLM reasoning by incorporating intermediate feedback during the reasoning process <cite class="ltx_cite ltx_citemacro_citep">(Lightman etÂ al., <a href="#bib.bib26" title="" class="ltx_ref">2024</a>; Ma etÂ al., <a href="#bib.bib27" title="" class="ltx_ref">2023</a>)</cite>. This approach has been shown to enhance the reasoning capabilities and accuracy of models in complex tasks <cite class="ltx_cite ltx_citemacro_citep">(Setlur etÂ al., <a href="#bib.bib34" title="" class="ltx_ref">2024</a>; Wu etÂ al., <a href="#bib.bib55" title="" class="ltx_ref">2024</a>)</cite>. Researchers have explored various methods to implement process supervision, including constructing intermediate supervision signals through rollouts, human feedback, or LLM feedback <cite class="ltx_cite ltx_citemacro_citep">(Zhang etÂ al., <a href="#bib.bib63" title="" class="ltx_ref">2024a</a>; Cobbe etÂ al., <a href="#bib.bib4" title="" class="ltx_ref">2021</a>; Wang etÂ al., <a href="#bib.bib48" title="" class="ltx_ref">2024a</a>; Sun etÂ al., <a href="#bib.bib39" title="" class="ltx_ref">2024</a>)</cite>. By shifting the focus from outcome-based rewards to reasoning trajectories, this paradigm fosters more nuanced understanding and improved problem-solving abilities in language models.</p>
</div>
</section>
</section>
<section id="S7" class="ltx_section">
<h2 class="ltx_title ltx_title_section">
<span class="ltx_tag ltx_tag_section">7 </span>Conclusion</h2>

<div id="S7.p1" class="ltx_para">
<p class="ltx_p">We presented RAG-Gym, a framework for optimizing reasoning and search agents through process supervision, and introduced ReSearch, an agent architecture that unifies answer reasoning with search query generation. Experiments show that RAG-Gym improves search agents on knowledge-intensive tasks, with ReSearch consistently outperforming baselines.
We also demonstrated the effectiveness of using LLMs as process reward judges, the transferability of trained reward models on different LLMs, and the scaling patterns of ReSearch during training and inference.</p>
</div>

<p class="ltx_p">Table <a href="#A1.T3" title="Table 3 â€£ Appendix A Comparison with Concurrent Work â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">3</span></a> provides a detailed comparison of RAG-Gym with concurrent methods. SmartRAG <cite class="ltx_cite ltx_citemacro_citep">(Gao etÂ al., <a href="#bib.bib9" title="" class="ltx_ref">2024</a>)</cite> shares a similar MDP formulation but relies solely on reinforcement learning-based tuning without process-level supervision. LeReT <cite class="ltx_cite ltx_citemacro_citep">(Hsu etÂ al., <a href="#bib.bib12" title="" class="ltx_ref">2024</a>)</cite>, AR-MCTS <cite class="ltx_cite ltx_citemacro_citep">(Dong etÂ al., <a href="#bib.bib6" title="" class="ltx_ref">2024</a>)</cite>, and CR-Planner <cite class="ltx_cite ltx_citemacro_citep">(Li etÂ al., <a href="#bib.bib24" title="" class="ltx_ref">2024</a>)</cite> incorporate process supervision but focus exclusively on either agent tuning or verifier training, without a unified framework for both. Additionally, these methods, along with the recent ReARTeR <cite class="ltx_cite ltx_citemacro_citep">(Sun etÂ al., <a href="#bib.bib40" title="" class="ltx_ref">2025</a>)</cite>, are tailored to specific agent architectures, limiting their generalizability. In contrast, RAG-Gym is agent-agnostic and supports both process reward modeling and direct agent tuning, enabling robust optimization of information-seeking agents. <section id="A2" class="ltx_appendix">
<h2 class="ltx_title ltx_title_appendix">
<span class="ltx_tag ltx_tag_appendix">Appendix B </span>Dataset Descriptions</h2>

<div id="A2.p1" class="ltx_para">
<p class="ltx_p">In this section, we provide detailed descriptions of the datasets used in our experiments, including HotpotQA <cite class="ltx_cite ltx_citemacro_citep">(Yang etÂ al., <a href="#bib.bib60" title="" class="ltx_ref">2018</a>)</cite>, 2WikiMultihopQA <cite class="ltx_cite ltx_citemacro_citep">(Ho etÂ al., <a href="#bib.bib11" title="" class="ltx_ref">2020</a>)</cite>, Bamboogle <cite class="ltx_cite ltx_citemacro_citep">(Press etÂ al., <a href="#bib.bib29" title="" class="ltx_ref">2023</a>)</cite>, and MedQA <cite class="ltx_cite ltx_citemacro_citep">(Jin etÂ al., <a href="#bib.bib17" title="" class="ltx_ref">2021</a>)</cite>.</p>
</div>
<section id="A2.SS0.SSS0.Px1" class="ltx_paragraph">
<h5 class="ltx_title ltx_title_paragraph">HotpotQA.</h5>

<div id="A2.SS0.SSS0.Px1.p1" class="ltx_para">
<p class="ltx_p">HotpotQA is a large-scale, multi-hop question-answering dataset that requires reasoning across multiple documents. It consists of questions that explicitly demand retrieving and synthesizing information from different sources. The dataset provides both distractor and supporting documents, allowing evaluation of modelsâ€™ ability to filter relevant information effectively. As the answers to the test questions in HotpotQA are not publicly available we took a subsample from its validation set (7,405 instances) as previous research did <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>; Li etÂ al., <a href="#bib.bib25" title="" class="ltx_ref">2025</a>)</cite>. The last 1,000 validation questions were selected for the agent evaluation on HotpotQA. The first 1,000 questions were used as the training data for process supervision.</p>
</div>
</section>
<section id="A2.SS0.SSS0.Px2" class="ltx_paragraph">
<h5 class="ltx_title ltx_title_paragraph">2WikiMultihopQA.</h5>

<div id="A2.SS0.SSS0.Px2.p1" class="ltx_para">
<p class="ltx_p">2WikiMultihopQA is another multi-hop question-answering dataset constructed from Wikipedia. 2WikiMultihopQA focuses on high-quality reasoning paths by selecting supporting documents more systematically. The dataset contains questions that require reasoning across different Wikipedia pages, ensuring a diverse range of factual and inferential challenges. The last 1000 questions in the development set (12,576 question in total) were used for agent evaluation.</p>
</div>
</section>
<section id="A2.SS0.SSS0.Px3" class="ltx_paragraph">
<h5 class="ltx_title ltx_title_paragraph">Bamboogle.</h5>

<div id="A2.SS0.SSS0.Px3.p1" class="ltx_para">
<p class="ltx_p">Bamboogle is a manually constructed dataset designed to evaluate compositional reasoning and adversarial robustness. It consists of 2-hop questions written by researchers, where both supporting facts exist in Wikipedia but are structured to be challenging for retrieval-based systems. Unlike automatically generated datasets like 2WikiMultihopQA and Musique, Bamboogle questions do not follow fixed templates, increasing their variability. We used the whole test set with 125 questions for the evaluation of agents on Bamboogle.</p>
</div>
</section>
<section id="A2.SS0.SSS0.Px4" class="ltx_paragraph">
<h5 class="ltx_title ltx_title_paragraph">MedQA.</h5>

<div id="A2.SS0.SSS0.Px4.p1" class="ltx_para">
<p class="ltx_p">MedQA is a medical question-answering dataset sourced from professional medical exams such as the USMLE (United States Medical Licensing Examination). It requires domain-specific knowledge and reasoning to answer multiple-choice medical questions. We focused on the English split of MedQA with 1,273 USMLE-style test questions. A subset of 1,000 questions was sampled from the training set (10,178 questions) for the optimization of various agents.</p>
</div>
</section>
</section>

<span class="ltx_tag ltx_tag_item">â€¢</span>

<p class="ltx_p">CoT <cite class="ltx_cite ltx_citemacro_citep">(Wei etÂ al., <a href="#bib.bib52" title="" class="ltx_ref">2022b</a>)</cite>: The agent outputs both the reasoning and the predicted answer at the first iteration.</p>
</div>
</li>
<li id="A3.I1.i3" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="A3.I1.i3.p1" class="ltx_para">

</div>
</li>
<li id="A3.I1.i4" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="A3.I1.i4.p1" class="ltx_para">
<p class="ltx_p">ReAct <cite class="ltx_cite ltx_citemacro_citep">(Yao etÂ al., <a href="#bib.bib61" title="" class="ltx_ref">2023</a>)</cite>: The agent reasons about the given state and generates either a search action or an answer action.</p>
</div>
</li>
<li id="A3.I1.i5" class="ltx_item" style="list-style-type:none;">
<span class="ltx_tag ltx_tag_item">â€¢</span> 
<div id="A3.I1.i5.p1" class="ltx_para">
<p class="ltx_p">Search-o1 <cite class="ltx_cite ltx_citemacro_citep">(Li etÂ al., <a href="#bib.bib25" title="" class="ltx_ref">2025</a>)</cite>: Before the state reasoning, the agent summarizes the retrieved documents in the information-seeking history as a direct answer to the corresponding search query. The agent uses query-answer pairs instead of query-documents pairs for the construction of LLM input as our ReSearch does. It can be considered a special version of ReAct with RAG instead of information retrieval (IR) as the tool.</p>

<section id="A4" class="ltx_appendix">
<h2 class="ltx_title ltx_title_appendix">
<span class="ltx_tag ltx_tag_appendix">Appendix D </span>Implementation Details</h2>

<div id="A4.p1" class="ltx_para">
<p class="ltx_p">For the implementation of the IR environment, we select Wikipedia as the supporting corpus for the retrieval of relevant information for questions from HotpotQA, 2WikiMultihopQA, and Bamboogle. For the environment of solving MedQA questions, we use a combination of medical textbooks and StatPearls which were pre-processed by <cite class="ltx_cite ltx_citemacro_citet">Xiong etÂ al. (<a href="#bib.bib57" title="" class="ltx_ref">2024</a>)</cite>. For all tasks, we used both lexical and semantic retrievers whose results were merged with Reciprocal Rank Fusion <cite class="ltx_cite ltx_citemacro_citep">(Cormack etÂ al., <a href="#bib.bib5" title="" class="ltx_ref">2009</a>)</cite>. BM25 <cite class="ltx_cite ltx_citemacro_citep">(Robertson etÂ al., <a href="#bib.bib32" title="" class="ltx_ref">2009</a>)</cite> and BGE-Base <cite class="ltx_cite ltx_citemacro_citep">(Xiao etÂ al., <a href="#bib.bib56" title="" class="ltx_ref">2023</a>)</cite> were used for HotpotQA, 2WikiMultihopQA, and Bamboogle, while in MedQA, we selected BM25 and MedCPT <cite class="ltx_cite ltx_citemacro_citep">(Jin etÂ al., <a href="#bib.bib18" title="" class="ltx_ref">2023</a>)</cite>.</p>
</div>
<div id="A4.p2" class="ltx_para">
<p class="ltx_p">For all LLM tuning in our paper, we employed the LoRA fine-tuning <cite class="ltx_cite ltx_citemacro_citep">(Hu etÂ al., <a href="#bib.bib13" title="" class="ltx_ref">2021</a>)</cite> with <math id="A4.p2.m1" class="ltx_Math" alttext="r=256" display="inline"><semantics><mrow><mi>r</mi><mo>=</mo><mn>256</mn></mrow><annotation encoding="application/x-tex">r=256</annotation></semantics></math> and <math id="A4.p2.m2" class="ltx_Math" alttext="alpha=512" display="inline"><semantics><mrow><mrow><mi>a</mi><mo lspace="0em" rspace="0em">â€‹</mo><mi>l</mi><mo lspace="0em" rspace="0em">â€‹</mo><mi>p</mi><mo lspace="0em" rspace="0em">â€‹</mo><mi>h</mi><mo lspace="0em" rspace="0em">â€‹</mo><mi>a</mi></mrow><mo>=</mo><mn>512</mn></mrow><annotation encoding="application/x-tex">alpha=512</annotation></semantics></math> on all attention components in the transformers architecture <cite class="ltx_cite ltx_citemacro_citep">(Vaswani, <a href="#bib.bib45" title="" class="ltx_ref">2017</a>)</cite>. The process supervision methods are implemented using the TRL package <cite class="ltx_cite ltx_citemacro_citep">(von Werra etÂ al., <a href="#bib.bib46" title="" class="ltx_ref">2020</a>)</cite>. For the tuning of Search-o1 and ReSearch agents, only the LLM for action reasoning is trained while the one for history knowledge summarization remains unturned.</p>

<section id="A5.SS1" class="ltx_subsection">
<h3 class="ltx_title ltx_title_subsection">
<span class="ltx_tag ltx_tag_subsection">E.1 </span>Comparison of Agent Architectures on Bamboogle</h3>

<div id="A5.SS1.p1" class="ltx_para">
<p class="ltx_p">We analyze the reasoning and search behaviors of RAG, ReAct, Search-o1, and ReSearch using an example from the Bamboogle dataset. As shown in Figure <a href="#A5.F6" title="Figure 6 â€£ E.1 Comparison of Agent Architectures on Bamboogle â€£ Appendix E Case Studies â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">6</span></a>, given the question â€œWhat was the father of the last surviving Canadian father of Confederation?â€, the three agents show distinct behaviors when generating the first action.</p>
</div>
<div id="A5.SS1.p2" class="ltx_para">
<p class="ltx_p">The RAG agent directly passes the question as a search query without decomposition, relying entirely on retrieval to infer the answer. This often leads to ineffective searches that fail to retrieve necessary intermediate facts. ReAct and Search-o1 improve upon this by engaging in stepwise query reasoning, first identifying the need to determine the last surviving Canadian father of Confederation before issuing a search query. However, the generated query, â€œList of Canadian fathers of Confederationâ€, retrieves broad information rather than directly resolving the missing knowledge.</p>
</div>
<div id="A5.SS1.p3" class="ltx_para">
<p class="ltx_p">In contrast, ReSearch explicitly integrates answer reasoning with search. It first constructs a potential answer, identifying an unverified claim that William Lyon Mackenzie King is among the last surviving Canadian fathers of Confederation. Recognizing the missing evidence, it formulates a targeted query, â€œWho is the last surviving Canadian father of Confederation?â€, to resolve the uncertainty. This approach ensures that retrieval is aligned with answer construction, minimizing unnecessary queries and improving information efficiency. The case study illustrates how ReSearch effectively refines the search process by linking query generation to specific knowledge gaps.</p>
</div>

<div id="A5.SS2.p1" class="ltx_para">
<p class="ltx_p">To illustrate how the Process Reward Modeling (PRM) improves decision-making, we examine a case from the MedQA dataset (Table <a href="#A5.T4" title="Table 4 â€£ E.2 PRM-Guided Action Selection in MedQA â€£ Appendix E Case Studies â€£ RAG-Gym: Optimizing Reasoning and Search Agents with Process Supervision" class="ltx_ref"><span class="ltx_text ltx_ref_tag">4</span></a>). The model is tasked with identifying the mechanism of action of the most likely anti-diabetic medication for a 60-year-old patient presenting with symptoms indicative of fluid overload.</p>
</div>
<div id="A5.SS2.p2" class="ltx_para">
<p class="ltx_p">For the generation of the first action, the agent initially generates a broad and less actionable query about heart failure, which, while relevant, does not directly contribute to identifying the correct medication. Meanwhile, it also provides another candidate action with the query â€œWhat are common classes of oral anti-diabetic medications?â€, which leads to retrieving a structured list of relevant drug categories. The process reward model estimates the quality of these two candidates and identifies the second candidate as the better action.</p>
</div>
<div id="A5.SS2.p3" class="ltx_para">
<p class="ltx_p">As the reasoning progresses, the process reward model continues to refine action selection, identifying the best queries to resolve missing information efficiently. The rewarded queries ultimately guide the agent toward correctly inferring that the medication most likely falls under the sulfonylureas class, which acts by closing ATP-sensitive K-channels in pancreatic beta cells.
This case demonstrates how process reward models enhance decision quality by selecting queries that effectively bridge knowledge gaps, leading to more precise reasoning and higher answer accuracy.</p>
