# TigerVector: Revolutionizing RAG with Vector Search Inside Your Graph Database

The explosion of Large Language Models (LLMs) has made Retrieval-Augmented Generation (RAG) a cornerstone for building accurate and reliable AI applications. By grounding LLMs with external data, RAG prevents hallucination and allows responses to be based on specific, up-to-date information.

Most RAG systems today rely heavily on vector databases, which store data as numerical embeddings that capture semantic meaning. This allows for efficient similarity search to find relevant information. However, real-world data is often not just a collection of isolated items; it's interconnected. Relationships between data points – like who wrote a document, which entities are mentioned, or how events are linked – are crucial for understanding context and answering complex queries. Pure vector search systems often struggle to capture these rich relationships, leading to suboptimal retrieval for nuanced questions.

This is where graph databases shine. They are designed to model and query relationships, representing data as nodes (entities) and edges (relationships) with properties. Leveraging knowledge graphs for RAG, a concept sometimes called GraphRAG, promises to retrieve context that is not only semantically relevant but also structurally connected. This is powerful for tasks like summarizing information across multiple linked documents or tracing dependencies.

But what if you need both? Many advanced RAG scenarios require both semantic similarity *and* relationship awareness. Imagine asking: "Find documents similar to this one that were written by someone connected to a specific project team." This requires combining vector search with graph traversal.

The straightforward approach is to use separate vector and graph databases. However, this introduces complexity: data silos, inconsistency between databases, data movement overhead, and managing two distinct query languages and access control systems. What's needed is a unified system.

Integrating vector search capabilities directly into a graph database is a compelling solution, but it presents significant technical challenges. Achieving high performance comparable to specialized vector databases, providing flexible query capabilities, handling transactional updates efficiently, and managing diverse types of vector embeddings within the graph structure are non-trivial problems. While some graph databases have started adding vector features, they often come with limitations in performance, functionality, or scalability.

### Introducing TigerVector

This is where TigerVector comes in. We present TigerVector, a new system seamlessly integrated into TigerGraph, a distributed Massively Parallel Processing (MPP) native graph database. TigerVector is designed to support VectorGraphRAG, a hybrid RAG approach that combines the power of vector search and graph search within a single, unified system.

TigerVector addresses the challenges of integrating vector search by:

1.  **Adding a First-Class `embedding` Type:** We introduce a dedicated `embedding` data type for vertex attributes. This is more than just a list of floats; it manages crucial metadata like dimensionality, model used, index type, and similarity metric. This allows for stricter compatibility checks during queries and supports multiple embedding types within the same vertex, essential for handling diverse data modalities (text, image, etc.). We also define the concept of an `embedding space` to group compatible embeddings across different vertex types.

    <br>![Figure 2: Example of Embedding Space](../test/tem/2501.11216_Figure2.png)<br>
    *Figure 2: Visualizing different Embedding Spaces and their association with vertex attributes.*

2.  **Decoupling Storage and Leveraging MPP:** To optimize performance and manage the large volume of vector data, vector embeddings are stored separately from other graph attributes, although they follow the same vertex-centric partitioning scheme. Each partition (segment) gets its own vector index (HNSW is currently supported, with pluggability for others). TigerVector fully utilizes TigerGraph's MPP architecture, distributing vector search tasks across machines and performing local segment searches followed by a global merge.

    <br>![Figure 3: Decoupled Storage](../test/tem/2501.11216_Figure3.png)<br>
    *Figure 3: Illustrating how vector embeddings for a vertex segment are stored separately in an embedding segment, but retain the same IDs as their corresponding vertices.*

3.  **Seamless GSQL Integration:** We extend TigerGraph's SQL-like query language, GSQL, to support declarative vector search using `ORDER BY VECTOR_DIST(...) LIMIT k`. More importantly, we introduce a flexible procedural function `VectorSearch()` that can take a vertex set as input (for filtering or compositional queries) and return a vertex set of results. This allows vector search to be easily combined with graph traversals and other GSQL features, enabling complex hybrid queries.

4.  **Efficient and Atomic Updates:** TigerVector reuses TigerGraph's MVCC (Multi-Version Concurrency Control) framework for transactional consistency. Vector updates (insertions, deletions, modifications) are initially recorded as deltas. Background vacuum processes efficiently flush these deltas to disk and incrementally merge them into the vector indexes. This decoupled vacuum process minimizes impact on foreground queries.

### Putting TigerVector to the Test

Our extensive experiments demonstrate that TigerVector is not just a conceptual integration but a high-performance system ready for demanding RAG workloads.

We compared TigerVector against Neo4j and Amazon Neptune (other graph databases with vector capabilities) and Milvus (a highly optimized specialized vector database) on standard vector search benchmarks (SIFT100M and Deep100M).

<br>![Figure 7: Throughput Evaluation on SIFT100M and Deep100M](../test/tem/2501.11216_Figure7.png)<br>
*Figure 7: Throughput comparison of TigerVector, Milvus, Neo4j, and Neptune Analytics on SIFT100M and Deep100M datasets across varying recall rates.*

TigerVector significantly outperformed Neo4j and Amazon Neptune in both throughput and recall, often by factors of 3x to 5x or more, while also being substantially more cost-effective than Neptune Analytics.

Perhaps most surprisingly, TigerVector achieved throughput comparable to, and sometimes even higher than, Milvus, a specialized vector database written in Go (TigerVector is in C++). This highlights the power of TigerGraph's underlying MPP architecture when leveraged for vector search. Latency results showed similar trends, with TigerVector being much faster than other graph databases and slightly faster than Milvus.

Beyond pure vector search, we evaluated TigerVector's scalability, demonstrating strong performance gains as nodes and data size increased. We also showed efficient index update performance, identifying a threshold beyond which full index rebuilds become more efficient than incremental updates.

Crucially, we demonstrated the power of hybrid search using modified LDBC SNB queries. These experiments showed that combining graph traversals with vector search for tasks like finding similar posts created by people within certain communities is not only possible but performs efficiently within TigerVector.

### Key Takeaways

Our experience building TigerVector reinforced several key lessons for integrating vector search into graph databases:

*   **Decoupling is Key:** Separating vector embeddings storage allows leveraging optimized vector index implementations and simplifies management and updates.
*   **MPP Matters:** Distributed architectures are crucial for scaling vector search performance on large graphs.
*   **Query Composition is Powerful:** Integrating vector search into a flexible graph query language enables expressing complex hybrid queries needed for advanced RAG.
*   **Unified Systems Deliver Value:** Combining vector and graph data management in one system addresses critical issues like consistency, data silos, and simplified governance.

These lessons are applicable beyond TigerGraph, suggesting that a unified approach is the path forward for databases supporting the evolving needs of AI applications like RAG.

### Conclusion

TigerVector is a significant step towards enabling advanced RAG and other AI applications that require leveraging both the semantic power of vector embeddings and the structural richness of graph data. By deeply integrating high-performance vector search into a distributed graph database, TigerVector provides a unified, scalable, and efficient platform that outperforms existing graph databases and holds its own against specialized vector systems. TigerVector was integrated into TigerGraph v4.2 in December 2024, making these capabilities available to users today.
